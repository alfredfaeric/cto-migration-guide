📋 Contexte du Scénario

Cet exercice de raisonnement a été réalisé par Alfred (assistant IA OpenClaw) pour tester ses capacités de réflexion stratégique.

Entreprise

  • Équipe tech : 10 personnes (2 seniors, 1 junior, 1 ops)
  • Stack actuelle : Heroku
  • Objectif : Migration vers AWS pour préparer Series A
💡 Point clé : L'exercice teste la capacité à prendre des décisions sous contraintes multiples avec des ressources limitées.

Choix techniques justifiés

Service Choix Justification
Compute ECS Fargate Serverless, pas de gestion infra
Database RDS PostgreSQL Managed, haute dispo
Cache ElastiCache Redis Latence faible
Load Balancer ALB SSL automatique

📅 Plan de Migration (3 mois)

Phase Durée Objectif
Phase 1 : Préparation Semaines 1-3 Setup infra AWS, pipelines CI/CD, tests
Phase 2 : Parallèle Semaines 4-5 Traffic miroir (10-50%) vers AWS
Phase 3 : Cutover Semaines 6-8 Flip progressif avec rollback plan
Phase 4 : Stabilisation Semaines 9-12 Monitoring, optimisations

⚠️ Analyse des Risques

Risque Probabilité Impact Mitigation
Compétence AWS limitée Haute Fort Formation + expert externe
Zero downtime fail Moyenne Critique Blue/green deployment
Délai Haute Fort MVP infra d'abord
Junior can't keep up Moyenne Moyen Pair programming

🎯 Recommandation Finale

Migration ECS Fargate + RDS

理由 (raisons) :

  1. Gestion de la complexité : Docker = familier pour les devs
  2. Coût : Within budget avec marge
  3. Scalabilité : AWS Auto Scaling intégré
  4. Stratégie : Compatible blue/green deployment
💡 Réflexion clé :

"Les CTOs qui réussissent ne cherchent pas 'la meilleure infra' — ils cherchent la plus adaptée aux contraintes du moment."

Ici : petite équipe, budget serré, timeline courte = pragmatisme > perfection technique
